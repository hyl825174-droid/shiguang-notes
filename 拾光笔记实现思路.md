# 拾光笔记（仿小红书）—— 全栈实现思路

> 基于 Spring Cloud Alibaba 微服务架构 + Vue 3 前端的社交内容平台

---

## 一、项目概述

拾光笔记是一个仿小红书的社交内容平台，核心功能包括：用户注册登录（短信验证码）、发布图文/视频笔记、点赞收藏、两级评论、关注粉丝、全文搜索等。采用前后端分离架构，后端为 Spring Cloud Alibaba 微服务，前端为 Vue 3 SPA。

### 4.2 评论表设计要点

评论采用**两级结构**，通过 `level` 字段区分：
- `level=1`：一级评论（直接评论笔记）
- `level=2`：二级评论（回复某条评论）

关键字段：
- `parent_id`：一级评论存储笔记 ID，二级评论存储一级评论 ID
- `reply_comment_id`：回复的目标评论 ID
- `reply_user_id`：回复的目标用户 ID
- `heat`：评论热度分数（用于排序）
- `child_comment_total`：二级评论总数

## 六、网关设计与路由

### 6.1 Gateway 路由配置

网关监听端口 `8000`，根据 URL 前缀路由到对应微服务（使用 `StripPrefix=1` 去掉前缀）：

| 路由前缀 | 目标服务 | 端口 |
|---------|---------|------|
| `/auth/**` | xiaohashu-auth | 8080 |
| `/user/**` | xiaohashu-user | 8082 |
| `/note/**` | xiaohashu-note | 8086 |
| `/relation/**` | xiaohashu-user-relation | 8087 |
| `/comment/**` | xiaohashu-comment | 8093 |
| `/oss/**` | xiaohashu-oss | 8081 |
| `/search/**` | xiaohashu-search | 8092 |

### 6.2 鉴权与身份传递

- **Sa-Token** 在网关层完成鉴权校验
- 校验通过后，将 `userId` 注入到 HTTP 请求头中
- 各微服务通过 `HeaderUserId2ContextFilter`（Servlet Filter）提取 `userId`，存入 `LoginUserContextHolder`（基于 `TransmittableThreadLocal`）
- Feign 调用时通过 `FeignRequestInterceptor` 自动传递 `userId` 请求头，形成完整的身份传播链

```
请求 → Gateway(Sa-Token 校验) → 注入 userId Header → 微服务 Filter 提取 → TTL 存储 → Feign 调用自动传递
```

## 七、核心业务实现思路

### 7.1 认证与登录

**实现方案：短信验证码 + Sa-Token**

```
┌────────┐    1.请求验证码     ┌──────────┐    调用阿里云 SMS    ┌──────────┐
│  前端   │ ─────────────────→ │ Auth服务  │ ─────────────────→ │ 阿里云SMS │
└────────┘                    └──────────┘                    └──────────┘
                                  │
                                  │ 2.验证码存入 Redis (key=手机号, TTL=3分钟)
                                  │    防刷：发送前检查 key 是否已存在
                                  │
┌────────┐    3.提交登录      ┌──────────┐    Feign调用     ┌──────────┐
│  前端   │ ─────────────────→ │ Auth服务  │ ─────────────→ │ User服务  │
└────────┘                    └──────────┘                  └──────────┘
                                  │                              │
                                  │ 4.Redis校验验证码             │ 5.查询/注册用户
                                  │ 5.Feign调用User服务           │
                                  │ 6.StpUtil.login(userId)      │
                                  │ 7.返回 Token                  │
```

**关键设计：**
- 验证码通过 `CompletableFuture` 异步发送，不阻塞主线程
- 密码使用 BCrypt 加密
- 支持两种登录方式：验证码登录、密码登录（策略模式切换）
- Token 有效期 30 天，风格为 `random-128`

### 7.2 笔记发布与详情

**笔记数据分离设计：**
- 笔记**元数据**（标题、类型、封面、频道 ID）存储在 MySQL `t_note` 表
- 笔记**正文内容**存储在 KV 存储服务（独立微服务），实现大文本与元数据的分离

**发布流程：**
```
前端提交笔记
    │
    ▼
NoteController.publishNote()
    │
    ├── 1. Feign调用分布式ID生成器获取雪花ID
    ├── 2. 处理话题（已有关联/新建话题）
    ├── 3. 保存笔记元数据到 MySQL
    ├── 4. Feign调用 KV 服务存储正文内容
    ├── 5. 初始化 Redis 计数数据
    └── 6. 发送 MQ 消息（用于搜索索引同步等）
```

**笔记详情查询——三级缓存：**

```
请求 → Caffeine 本地缓存 → 命中? → 返回
              │ 未命中
              ▼
         Redis 缓存 → 命中? → 写入 Caffeine → 返回
              │ 未命中
              ▼
         MySQL 查询 → 写入 Redis + Caffeine → 返回
              │ 未命中
              ▼
         缓存空对象 (TTL=60s+随机) 防止缓存穿透
```

**缓存击穿/雪崩/穿透防护：**
- 所有缓存 TTL 添加随机偏移（如 `86400 + random(86400)`）防止雪崩
- 空结果缓存短 TTL 防止穿透
- 通过 RocketMQ **广播消息**通知所有实例清除本地 Caffeine 缓存

**并行数据组装：**
笔记详情需要组装用户信息、内容、计数三类数据，使用 `CompletableFuture.allOf()` 并行发起 3 个 Feign RPC 调用。

### 7.3 点赞/收藏（去重与计数）

**核心难点：高并发下的去重判断**

采用 **Roaring Bitmap + ZSet 双重数据结构**方案：

```
┌──────────────────────────────────────────────────────┐
│                   点赞/收藏架构                        │
│                                                      │
│  1. 判断是否已点赞/收藏                                │
│     → Roaring Bitmap (Lua 脚本原子操作)               │
│     → 概率型快速判断，空间效率极高                      │
│                                                      │
│  2. 存储点赞/收藏列表                                  │
│     → Redis ZSet (score=时间戳)                       │
│     → 支持按时间排序、分页查询                          │
│                                                      │
│  3. 懒初始化                                          │
│     → 首次访问时从 DB 批量加载到 Bitmap + ZSet         │
│                                                      │
│  4. 异步持久化                                        │
│     → MQ 有序消息 (asyncSendOrderly, hashKey=userId)  │
│     → 保证同一用户的操作顺序性                          │
│     → Count 服务消费后聚合写入 DB                      │
└──────────────────────────────────────────────────────┘
```

**Lua 脚本保证原子性：** 点赞/取消点赞操作通过 Lua 脚本在一次 Redis 调用中完成"检查是否已操作 + 添加/移除记录"。

### 7.4 评论系统

**两级评论模型：**

```
笔记详情
  ├── 一级评论 (level=1, parent_id=笔记ID)
  │     ├── 二级回复 (level=2, parent_id=一级评论ID, reply_comment_id=目标评论ID)
  │     ├── 二级回复 ...
  │     └── first_reply_comment_id → 最早的一条回复（预览用）
  ├── 一级评论 ...
  └── ...
```

**评论排序与缓存：**
- 一级评论按 `heat` 热度分数排序（Redis ZSet，`ZREVRANGEBYSCORE` 分页）
- 二级回复按时间排序（Redis ZSet）
- 热门评论批量同步到 Redis（每批 500 条），仅缓存前 50 页一级评论和前 10 页二级回复
- 评论详情（JSON）缓存到 Redis String

**评论点赞去重：** 使用 **Redis Bloom Filter**（Lua 脚本），懒初始化，首次访问时从 DB 加载。

**评论热度计算：** `HeatCalculator` 基于点赞数、回复数等因素计算热度分数。

### 7.5 关注/粉丝系统

**Redis ZSet + Lua 脚本方案：**

```
┌─────────────────────────────────────────┐
│            关注/粉丝架构                  │
│                                         │
│  Redis ZSet 存储关系                     │
│  ├── following:{userId}  → 关注列表     │
│  │   (score=时间戳, value=目标用户ID)    │
│  └── fans:{userId}       → 粉丝列表     │
│      (score=时间戳, value=粉丝用户ID)    │
│                                         │
│  Lua 脚本原子操作                        │
│  ├── follow_check_and_add.lua           │
│  │   → 检查是否已关注                    │
│  │   → 限制关注上限 1000 人              │
│  │   → 同时添加关注+粉丝记录             │
│  └── unfollow_check_and_delete.lua      │
│      → 检查关注关系是否存在               │
│      → 同时删除关注+粉丝记录             │
│                                         │
│  懒初始化：ZSet 不存在时从 DB 全量加载    │
│  粉丝列表最多缓存 5000 条                │
│                                         │
│  MQ 异步持久化到 MySQL                   │
│  → t_following 表 + t_fans 表            │
└─────────────────────────────────────────┘
```

**有序 MQ：** 使用 `asyncSendOrderly` 并以 `userId` 为 hash key，保证同一用户的关注/取关操作按顺序执行。

### 7.6 计数服务（聚合流水线）

**核心挑战：** 高并发下的计数（点赞数、粉丝数、评论数等）如何高效更新？

**两阶段 MQ 聚合方案：**

```
┌──────────┐    MQ消息      ┌──────────────────────┐
│ Note服务  │ ─────────────→│ Stage 1: 消费者       │
│ (点赞等)  │               │ BufferTrigger 微批    │
└──────────┘               │ (容量5万/批1000/等待1s)│
                           └──────────┬───────────┘
                                      │ 按 noteId 分组聚合
                                      ▼
                           ┌──────────────────────┐
                           │ Stage 2: 更新 Redis   │
                           │ HINCRBY Hash 计数器   │
                           └──────────┬───────────┘
                                      │ 发送聚合结果
                                      ▼
                           ┌──────────────────────┐
                           │ Stage 3: 持久化到 DB  │
                           │ 批量更新 t_note_count │
                           └──────────────────────┘
```

**BufferTrigger（快手开源）：** 提供微批处理能力，将高频单条消息聚合为低频批量操作，极大降低 DB 写入压力。

**Redis Hash 存储计数：**
- 笔记计数 Hash：`like_total`、`collect_total`、`comment_total`
- 用户计数 Hash：`like_total`、`fans_total`、`following_total`、`note_total`

### 7.7 搜索服务（Elasticsearch + Canal）

**Canal CDC 方案实现近实时搜索索引：**

```
┌──────────┐   Binlog    ┌───────────────────┐
│  MySQL   │ ──────────→ │ Canal Server       │
│ (t_note, │             │ (监听 Binlog 变更)  │
│  t_user) │             └─────────┬─────────┘
└──────────┘                       │
                                   │ 100ms 轮询
                                   ▼
                         ┌───────────────────┐
                         │ Search 服务        │
                         │ CanalSchedule      │
                         │                   │
                         │ INSERT → 索引文档  │
                         │ UPDATE → 更新/删除 │
                         │ DELETE → 移除文档  │
                         └─────────┬─────────┘
                                   │
                                   ▼
                         ┌───────────────────┐
                         │ Elasticsearch     │
                         │ (笔记索引 + 用户索引)│
                         └───────────────────┘
```

**搜索查询设计：**
- `multi_match` 查询：标题权重 2.0，话题权重 1.0
- 综合排序使用 `function_score`：`like_total × sqrt(0.5) + collect_total × sqrt(0.3) + comment_total × sqrt(0.2)`
- 搜索结果高亮：`<strong>` 标签包裹匹配词
- 支持按类型、发布时间范围过滤
- 使用 IK Analyzer 中文分词 + 自定义扩展词典

### 7.8 分布式 ID 生成（美团 Leaf 方案）

提供**双策略** ID 生成能力：

**策略一：Snowflake（雪花算法）**
- 基于 ZooKeeper 分配 workerId（最大 1023）
- 12 位序列号，每毫秒可生成 4096 个 ID
- 时钟回拨处理：回拨 ≤ 5ms 则等待，否则报错
- 序列号从随机值（0-99）开始，降低可预测性
- `synchronized` 保证线程安全

**策略二：Segment（号段模式）**
- 双 Buffer 策略：当一个号段消耗到阈值时，异步预加载下一个号段
- 基于 DB 号段表，每次获取一个范围的 ID
- 适合不需要严格递增但需要高可用的场景

### 7.9 数据对齐服务（最终一致性保障）

**核心问题：** Redis 中的计数与 MySQL 中的计数可能不一致，需要定期对账修正。

**分片定时任务方案：**

```
┌────────────────────────────────────────────────────────┐
│                   数据对齐流程                           │
│                                                        │
│  1. MQ 消费者将增量数据写入临时分片表                     │
│     → t_data_align_note_like_count_temp_20240520_0     │
│     → 按日期 + 分片索引分表                              │
│                                                        │
│  2. XXL-Job 定时任务触发对齐                              │
│     → 使用 XxlJobHelper.getShardIndex() 分片           │
│     → 每次处理 1000 条                                  │
│                                                        │
│  3. 对齐逻辑                                            │
│     → 读取临时表中的增量记录                              │
│     → 对源表执行 COUNT(*) 获取真实计数                   │
│     → 更新计数表 + Redis 缓存                           │
│     → 删除已处理的临时记录                                │
│     → 调用 Search 服务重建 ES 文档                       │
│                                                        │
│  4. 表生命周期管理                                       │
│     → CreateTableXxlJob: 每日创建新临时表               │
│     → DeleteTableXxlJob: 清理过期临时表                 │
└────────────────────────────────────────────────────────┘
```

**7 类对齐任务：**
- 笔记维度：点赞计数、收藏计数、发布计数
- 用户维度：关注计数、粉丝计数、获赞计数、收藏计数

## 九、关键技术方案总结

### 9.1 缓存策略

| 策略 | 实现 |
|------|------|
| **三级缓存** | Caffeine（本地）→ Redis → MySQL |
| **防缓存穿透** | 空结果缓存（TTL 60s + 随机值） |
| **防缓存雪崩** | TTL 添加随机偏移（基础时间 + random(基础时间)） |
| **缓存一致性** | MQ 广播消息通知所有实例清除本地缓存 |
| **懒加载** | Redis 数据不存在时从 DB 批量加载 |

### 9.2 高并发去重

| 场景 | 方案 |
|------|------|
| 笔记点赞/收藏 | Roaring Bitmap（概率型快速判断）+ ZSet（精确列表） |
| 评论点赞 | Bloom Filter（Lua 脚本）+ DB 回查 |
| 关注关系 | Redis ZSet + Lua 脚本（原子检查+添加，限制上限 1000） |

### 9.3 异步与消息

| 场景 | 方案 |
|------|------|
| 点赞/收藏持久化 | RocketMQ 有序消息（per-user 顺序保证） |
| 计数聚合 | BufferTrigger 微批 + 两阶段 MQ 管线 |
| 搜索索引同步 | Canal 监听 Binlog（100ms 轮询） |
| 缓存失效 | RocketMQ 广播消息 |
| 数据对齐 | XXL-Job 分片定时任务 + 临时分片表 |

### 9.4 分布式方案

| 场景 | 方案 |
|------|------|
| 唯一 ID | Snowflake（ZK 分配 workerId）+ Segment（双 Buffer 号段） |
| 用户身份传播 | Gateway 提取 → HTTP Header → Servlet Filter → TTL → Feign Interceptor |
| 服务注册发现 | Nacos |
| 配置管理 | Nacos Config |
| 分布式调度 | XXL-Job（分片广播） |

### 9.5 数据一致性保障

```
实时写入 Redis → MQ 异步写入 DB → 定时对账修正
                                        │
                                        ▼
                              最终一致性（秒级延迟）
```

- **强一致性**：Redis 操作（Lua 脚本保证原子性）
- **最终一致性**：DB 持久化通过 MQ 异步完成
- **对账修正**：数据对齐服务通过定时任务定期校正 Redis 与 DB 的差异

## 十二、项目亮点

1. **三级缓存架构**：Caffeine + Redis + MySQL，配合随机 TTL 防雪崩、空缓存防穿透、MQ 广播防不一致
2. **Roaring Bitmap 去重**：高并发点赞/收藏场景下的高效去重方案
3. **两阶段计数聚合**：BufferTrigger 微批 + MQ 管线，将 DB 写入压力降低数个量级
4. **Canal CDC 搜索同步**：应用层无需关心索引维护，Binlog 监听自动同步 ES
5. **分片数据对齐**：XXL-Job 分片 + 临时分片表，实现可扩展的数据一致性校验
6. **双策略分布式 ID**：Snowflake + Segment 双保险，适配不同业务场景
7. **完整的前后端分离**：前端 Vue 3 现代化架构，后端微服务独立部署
