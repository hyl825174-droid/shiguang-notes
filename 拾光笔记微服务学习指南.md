# 拾光笔记 —— 微服务实践学习指南

> 基于拾光笔记项目，快速高效地实践掌握微服务核心技术

---

## 一、核心学习思路：学架构，不学业务

逐行敲整个项目效率很低，因为大量代码是业务 CRUD 和 UI 细节，和微服务核心知识关系不大。

**真正要掌握的微服务知识：**

| 知识点 | 本项目对应位置 |
|--------|--------------|
| 服务注册与发现 | Nacos |
| 服务间调用 | OpenFeign |
| 统一网关 + 鉴权 | Gateway + Sa-Token |
| 分布式缓存策略 | Redis + Caffeine 三级缓存 |
| 异步解耦 | RocketMQ |
| 分布式搜索 | Elasticsearch + Canal |
| 分布式 ID | Snowflake / Segment |
| 数据一致性 | BufferTrigger + 对齐任务 |

> 笔记的 CRUD、前端页面、表单校验这些对学微服务帮助很小，可以跳过。

## 二、分阶段递进实践

### 第一阶段：跑通骨架（理解整体）

**目标：** 让 3 个最简单的服务通过网关调通，理解请求链路。

只需要搭建：

```
Gateway(8000) → Auth(8080) → User(8082)
```

**实践步骤：**

1. 搭建 Nacos，创建一个空的 `xiaoha-common` 公共模块
2. 搭建 Gateway，配置一条到 Auth 的路由规则
3. 搭建 Auth 服务，写一个 `/login` 接口（先不接短信，写死返回 Token）
4. 搭建 User 服务，写一个 `/user/profile` 接口
5. Auth 通过 Feign 调用 User（模拟登录时查询用户）
6. 用 Postman/curl 走通整个链路

**学到的：** Nacos 注册、Gateway 路由、Feign 调用、Maven 多模块管理

> 这个阶段不用抄代码，直接对着项目结构自己写，写不出来再看原项目。

---

### 第二阶段：鉴权 + 用户身份传播（理解微服务的认证难题）

**目标：** 理解 Token 如何在多个服务之间传递。

**只关注这几个文件：**

```
Gateway 网关层 Sa-Token 校验
  → HeaderUserId2ContextFilter（提取 userId 到 ThreadLocal）
    → LoginUserContextHolder（TTL 存储）
      → FeignRequestInterceptor（Feign 调用时传递 Header）
```

**实践：**

1. 在 Gateway 集成 Sa-Token，未登录请求拦截
2. 登录成功后，网关在请求头塞 `userId`
3. 各微服务写一个 Filter 从 Header 取 `userId`，存到 ThreadLocal
4. 写一个 Feign Interceptor，自动把 `userId` 带到下游服务
5. 验证：A 服务 → B 服务 → C 服务，`userId` 全程可获取

**学到的：** 微服务鉴权、ThreadLocal 传播、Feign 拦截器、TransmittableThreadLocal

---

### 第三阶段：选一个"高价值"模块深挖

从以下三个方向中**选一个**做到底：

#### 方向 A：笔记点赞（学习缓存 + MQ + 去重）—— 最推荐

最值得学的部分，包含大量微服务核心技术：

```
只需要实现：
1. Note 服务：发布笔记、查询笔记详情（三级缓存）
2. 点赞功能：Redis Lua 脚本去重 + ZSet 存储
3. Count 服务：RocketMQ 消费点赞消息 → BufferTrigger 聚合 → 批量写 DB
```

**不要抄所有代码**，只实现核心链路：

- 写一个简化版 Lua 脚本（检查 + 添加）
- 写一个 BufferTrigger 消费者（接收点赞消息，计数 +1）
- 验证：Redis 计数准不准，重启后 DB 数据对不对

#### 方向 B：搜索同步（学习 Canal + ES）

```
1. 搭建 ES + Canal
2. 写一个定时任务轮询 Canal，监听 t_note 表变更
3. 将变更同步到 ES 索引
4. 写一个搜索接口，用 multi_match 查询
```

#### 方向 C：分布式 ID（学习 Snowflake + ZK）

```
1. 搭建 ZK
2. 实现 SnowflakeIDGenImpl（workerId 从 ZK 获取）
3. 实现 Segment 双 Buffer（号段模式）
4. 对比两种方案的优缺点
```

---

### 第四阶段：拼装完整链路

把前三个阶段的成果串联起来，跑通一个完整的业务流程：

```
用户登录 → 发布笔记 → 搜索到笔记 → 点赞 → 计数更新
```

这时候你会真正理解：一个请求是怎么跨多个服务协作完成的。

## 三、加速技巧

### 3.1 用脚手架快速搭建模块骨架

不用手写每个模块的 pom.xml、启动类、application.yml，写一个脚本或模板：

```bash
# 新建一个微服务模块，只需要改三个东西：
# 1. pom.xml 的 artifactId
# 2. Application 启动类名
# 3. application.yml 的端口号
```

### 3.2 先跑通再优化

| 第一步（能跑就行） | 第二步（学习技术） |
|-------------------|-------------------|
| 点赞直接写 MySQL | 改为 Redis + MQ 异步 |
| 笔记详情直接查 DB | 加三级缓存 |
| 搜索直接查 MySQL | 换成 Canal + ES |
| ID 用数据库自增 | 换成 Snowflake |

> 每一轮迭代你都在学一个新技术，而且能清楚看到改进效果。

### 3.3 只抄关键代码，其余自己写

**值得逐行理解并手敲的代码：**

- Gateway 路由配置 + Sa-Token 集成
- Lua 脚本（点赞/关注去重）
- BufferTrigger 消费者
- Canal 数据同步逻辑
- Feign 拦截器
- Snowflake ID 生成核心逻辑

**直接复制或跳过的代码：**

- 数据库建表语句
- MyBatis Mapper XML
- 前端所有页面和组件
- DTO/VO 定义
- 通用工具类

### 3.4 画架构图辅助理解

每做完一个阶段，画一个时序图或架构图，比如：

```
点赞请求的完整链路：

前端 → Gateway → Note服务(Redis Lua去重) → RocketMQ → Count服务(BufferTrigger聚合) → MySQL
                                                           ↓
                                                      Redis HINCRBY
```

画图比看代码更能加深对整体设计的理解。

## 四、学习时间规划

```
阶段1: 跑通骨架（Gateway + Nacos + Feign）          ← 1-2天
         ↓
阶段2: 鉴权 + 身份传播（Sa-Token + ThreadLocal）     ← 1天
         ↓
阶段3A: 点赞缓存 + MQ                              ← 2-3天（最推荐）
  或 3B: Canal + ES 搜索同步                        ← 2天
  或 3C: 分布式 ID 生成                              ← 1天
         ↓
阶段4: 串联完整链路                                  ← 1天
```

总计约 **5-8 天**，就能把微服务最核心的知识都实践一遍。

**关键是每个阶段都要自己动手验证，而不只是看代码。**

## 六、验证清单

每个阶段完成后，用以下方式验证你是否真正掌握了：

### 第一阶段验证

- [ ] 能在 Nacos 控制台看到所有注册的服务
- [ ] 通过 Gateway 端口能访问到后端接口
- [ ] 关掉 Nacos，服务调用失败（证明服务发现生效）
- [ ] 能说清一个 HTTP 请求从浏览器到数据库的完整链路

### 第二阶段验证

- [ ] 不带 Token 请求被 Gateway 拦截，返回 401
- [ ] 登录获取 Token 后，请求能正常到达业务服务
- [ ] 在任意业务服务中能通过 `LoginUserContextHolder` 获取当前用户 ID
- [ ] A 服务通过 Feign 调 B 服务时，B 服务也能拿到用户 ID

### 第三阶段A 验证

- [ ] 第一次查询笔记详情时走数据库（看日志），第二次走缓存
- [ ] 重启服务后本地缓存失效，但 Redis 缓存还在
- [ ] 同一用户对同一笔记重复点赞，计数不会重复增加
- [ ] 点赞后 Redis 计数立刻 +1，DB 计数延迟但不丢数据
- [ ] 能解释为什么用 Roaring Bitmap 而不是 Set 做去重

### 第三阶段B 验证

- [ ] 插入一条笔记到 MySQL，ES 中能搜到
- [ ] 更新笔记标题，搜索结果同步更新
- [ ] 删除笔记，搜索结果消失
- [ ] 能解释 Canal 监听 Binlog 的原理

### 第三阶段C 验证

- [ ] 并发调用 ID 生成接口，不会产生重复 ID
- [ ] 重启服务后 ID 继续递增，不会回退
- [ ] 能说清 Snowflake 和 Segment 两种方案的适用场景

## 八、学习心态建议

1. **不要追求完美** —— 第一版能跑就行，后续迭代优化才是学习的过程
2. **多画图** —— 每理解一个流程就画一张时序图，比看十遍代码管用
3. **多断点调试** —— 跟着请求走一遍完整链路，比看文章理解更深
4. **记录问题** —— 遇到的问题和解决方案记下来，这是最宝贵的学习成果
5. **对比思考** —— 每学一个技术方案，想想"为什么不用更简单的方案？"理解取舍
